# Discord Macro Activity Bot

## Overview

Sistema de monitoramento de atividade para comunidades no Discord onde usuários utilizam macros externos em um jogo e enviam eventos através de webhooks.

O objetivo do sistema é:
- medir atividade real dos usuários
- classificar usuários em estados de atividade
- garantir cumprimento de quotas mínimas
- aplicar recompensas/punições automáticas
- fornecer ferramentas administrativas por servidor

O sistema é multi-guild, porém cada guild opera de forma completamente isolada.

---

# Product Vision

Criar comunidades auto-gerenciáveis onde membros precisam manter atividade mínima diária/semanal/mensal.

Caso o usuário:
- mantenha atividade suficiente → recebe benefícios
- fique inativo → perde benefícios
- entre em estado idle → sistema acompanha degradação de status
Toda automação é feita através do bot.

---

# Core Workflow

Fluxo principal:

1. Game biome changes
2. User Macro detects biome
3. User Macro sends notification through webhook
[ON OUR CONTROL]
4. Bot parses incoming message
5. Bot identifies user
6. Bot calculates active time delta
7. Bot updates user activity state (biomes, activity, other infos)
8. Background workers process quotas + roles

# Guild Setup Flow

Cada usuário possui canal próprio.
Fluxo:
```
User joins guild
    ↓
User runs /bh setup
    ↓
Bot finds available category
    ↓
Bot creates private channel
    ↓
Bot creates Discord webhook
    ↓
Bot sends webhook URL to user via DM
    ↓
User configures macro with webhook
    ↓
User starts macroing....
    ↓
User checks profile, etc... from here onwards he is managed by the bot
```

Observação:

Guild define categorias permitidas para macro. (via comando de configuração)
Exemplo:
```
allowed_categories = [
    category_1,
    category_2,
    category_3,
    category_4
]
```

Se limite de canais for atingido (Discord limita 50 canais por categoria):
- **Comportamento default (`auto_create_categories = false`):** setup é negado, bot avisa administrador.
- **Comportamento opcional (`auto_create_categories = true`):** bot cria automaticamente uma nova categoria e a adiciona à lista de `allowed_categories` da guild, sem intervenção manual do admin.

A flag é configurável por guild e vem **desabilitada por padrão**.

# Activity Engine

Atividade NÃO é heartbeat contínuo.
Cada mensagem recebida representa um evento discreto.

Algoritmo:

```
delta = now - last_activity

if delta <= activity_threshold:
    active_time += delta

else:
    session expired
    start new session

last_activity = now
```

Exemplo:

```
threshold = 20min
10:00 message
10:12 message → +12m
10:25 message → +13m
11:10 message
45min > threshold
ignore delta
new session starts
```

# Status Engine
Usuários possuem 3 estados.

## ACTIVE
Usuário enviando eventos normalmente.

## IDLE
Tempo sem atividade ultrapassou limite configurado.

## INACTIVE
Usuário ficou ausente por tempo excessivo.

Se o usuário ficar inativo por muito tempo, pode ou não (baseado em configuração de guilda) haver a exclusão dos dados do usuário. Essa ação pode, se configurada, ser prevenida por-user se um administrador usar um comando para manualmente pausar o canal do usuário.

State machine:

```
ACTIVE → IDLE → INACTIVE
INACTIVE → ACTIVE or DELETED/SUSPENDED
IDLE → ACTIVE
```

Scheduler:

```
Every N minutes:

if now - last_activity > idle_threshold:
    ACTIVE → IDLE

if now - last_activity > inactive_threshold:
    IDLE → INACTIVE
```

# Quota System

Sistema baseado em rolling windows.
Sem resets globais.

Motivo:
Evitar mass updates simultâneos e Discord rate limit.

Tipos de quota:
- daily (últimas 24h)
- weekly (últimos 7 dias)
- monthly (últimos 30 dias)
- configurable (administrador pode configurar a rolling-window que desejar)

O administrador pode escolher a rolling window que desejar utilizar.

Exemplo:

```
daily_required = 6h

if activity_24h >= 6h:
    compliant = true
```

**Nota de implementação:** cálculo de `activity_24h`/`7d`/`30d` via `SUM` direto sobre `activity_sessions` é aceitável para o MVP. Caso a performance degrade em produção (muitos usuários / verificações frequentes), avaliar migração para agregação incremental (saldo atualizado a cada sessão fechada, em vez de recomputado do zero a cada checagem). Não é bloqueante para as sprints iniciais — revisar no Sprint 10 se necessário.

---

# Reward System

Quando quota é cumprida:
- usuário recebe role active
- usuário mantém benefícios

Quando quota não é cumprida:
- remove role active
- remove acesso a benefícios

Usuário continua acumulando atividade mesmo após cumprir quota.

Motivo:
Permitir visualização contínua do risco de perder status.

## activity_events

Eventos processados (sem payload bruto — apenas os campos que o Activity/Status Engine consome).

```
id
user_id
discord_message_id   -- UNIQUE, usado para idempotência/dedup na ingestão
biome
macro_type
event_timestamp       -- timestamp reportado pelo macro, se houver
received_at
```

> Removido `raw_payload`. Não há necessidade de persistir o JSON completo, apenas o state derivado. `discord_message_id` (id da mensagem de webhook no Discord) é único por natureza e garante idempotência sem necessidade de hash customizado — mensagens duplicadas/reenviadas são rejeitadas por constraint `UNIQUE` na ingestão.
